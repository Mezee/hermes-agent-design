# Control Room Patterns

Use the control room as governance, not as the place where most work happens.

## What The Control Room Is For

- documenting which agents exist
- recording scope, credentials, and ownership
- storing runbooks and setup instructions
- tracking which workflows are in prototype, specialist, or automated stages
- keeping environment maps and operating notes

## What The Control Room Is Not For

- day-to-day execution of every task
- hiding operational state from the actual agents
- replacing specialist prompts or workspaces
- becoming a second product surface the user must manage manually

## Good Pattern

- user talks to a specialist when they know who owns the job
- user talks to the orchestrator when they want one front door
- operators use the control room for fleet configuration, governance, and runbooks

## Review Questions

- does the control room document the fleet clearly
- are agent responsibilities visible there
- would work still happen if the control room were read-only during execution
- is the control room simplifying governance instead of stealing the workflow surface
